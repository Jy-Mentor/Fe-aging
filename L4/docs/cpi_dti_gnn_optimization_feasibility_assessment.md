## 一、评估目标与范围

### 1.1 评估目标

本报告针对 `cpi_dti_gnn_optimization_research_report_v2.md` 中提出的架构升级、特征工程、训练策略与评估体系改进方案，进行系统化落地可行性评估。评估核心目标是在当前代码架构与硬件约束下，判断各改进方案能否安全、有效地整合，避免引入新的技术债务或功能异常。

### 1.2 评估范围

- **P1：HGT 工程修复**（解码器输入对齐、线性投影统一维度、HGTLoader 2 阶子图采样、全局索引、mini-batch 验证）
- **P2：ESM-2 150M 残基级特征重构**（`[L, 640]` 输入 `ResidueAwareBilinearDecoder`）
- **P3：双线性/跨模态注意力**（DrugBAN 风格 BANLayer 适配）
- **P4：预训练分子语言模型**（ChemBERTa-2 / MolFormer SMILES 嵌入）
- **P5：异质图网络扩展**（drug-drug 相似性、疾病关联等辅助关系）
- **训练策略**：课程负采样、DDB、拓扑/ESM-2 难负样本、Focal+BPR 损失
- **评估体系**：AUPR、Precision\@K、EF\@1%、EF\@5%、ROCE 等指标

#### 关键接口影响

1. **`HGTLinkPredictor.decode()`**：当前签名已支持 `prot_residue_indices`，但验证流程未强制传递全局索引。落地后需确保所有调用点均传递全局索引。
2. **`sample_hetero_subgraph()`**：`add_seed_cpi_edges` 参数必须在训练阶段关闭，否则造成信息泄漏。若保留该参数，需增加断言或日志警告。
3. **`ResidueAwareBilinearDecoder.register_residue_buffers()`**：当前使用普通属性保存大张量，已正确处理 `model.to(device)` 不搬迁。落地时需保持该设计。

- 可插拔解码器架构（MLP/Dot/Bilinear/ResidueAwareBilinear）已存在。

- **P5**：新边类型的 metadata 声明和采样逻辑缺失。
- **评估指标**：AUPR 已具备，但 Precision\@K、EF\@1%、EF\@5%、BEDROC、ROCE 需补充实现。

## 三、内存与性能评估

### 3.1 显存占用估算（RTX 5060 8GB）

| 方案                             | 估算显存占用        | OOM 风险 | 触发场景                                               |
| ------------------------------ | ------------- | ------ | -------------------------------------------------- |
| **P1：HGT 工程修复**                | +0.3\~0.8 GB  | 低      | HGTLoader 引入额外邻居缓存，但在 2 阶小采样下可控。                   |
| **P2：ESM-2 残基** **`[L, 640]`** | +2.0\~4.0 GB  | **高**  | 完整残基张量加载、max\_len=1024、batch 过大。                   |
| **P3：双线性交叉注意力**                | +0.2\~0.8 GB  | 中      | 注意力矩阵规模为 `[N, L, L]`，max\_len 增大时显存平方增长。           |
| **P4：ChemBERTa-2/MolFormer**   | +1.5\~2.5 GB  | 中      | 预训练模型参数和 SMILES 编码同时驻留 GPU。                        |
| **P5：异质图扩展**                   | +0.1\~0.5 GB  | 低      | 新增边和节点嵌入占用，通常可控。                                   |
| **基线模型（当前）**                   | \~2.5\~3.5 GB | 低      | hidden\_dim=128、num\_layers=2、batch\_size=128/256。 |

### 3.2 CPU RAM 需求评估

| 方案 | CPU RAM 需求   | 说明                                                                 |
| -- | ------------ | ------------------------------------------------------------------ |
| P1 | \~2 GB       | 图采样结构、邻居索引临时数据。                                                    |
| P2 | **\~10 GB+** | 完整 ESM-2 残基特征文件 8.86 GB，加上 Python/PyTorch 开销需 10 GB 以上。当前机器已触发段错误。 |
| P3 | \~1 GB       | 注意力中间矩阵主要驻留 GPU，CPU 侧为索引和掩码。                                       |
| P4 | \~2 GB       | 预训练模型参数加载和 SMILES 缓存。                                              |
| P5 | \~0.5 GB     | 新增边表和节点特征。                                                         |

### 3.3 训练与推理延迟估算

| 方案 | 训练时间影响    | 推理延迟影响                         |
| -- | --------- | ------------------------------ |
| P1 | +10\~20%  | +5\~10%（mini-batch 验证引入额外采样开销） |
| P2 | +50\~100% | +30\~50%（残基注意力逐残基计算）           |
| P3 | +20\~40%  | +15\~25%（交叉注意力额外计算）            |
| P4 | +20\~30%  | +10\~20%（SMILES 编码不可忽略）        |
| P5 | +5\~15%   | +5%（更多边类型消息传递）                 |

### 3.4 OOM 规避策略

1. **残基特征 CPU Offload**：保持 `ResidueAwareBilinearDecoder` 将 `_residue_embeddings` 作为普通属性存 CPU，仅在前向时按需搬迁当前 batch 到 GPU。
2. **分块前向**：维持 `max_residue_batch=8` 或更小（如 4），控制同时处理的蛋白-化合物对数量。
3. **降低 max\_len**：将默认 `max_len` 从 1024 降至 512 或 256，可显著降低残基张量显存占用（降至 1/4\~1/16）。
4. **混合精度训练（AMP）**：已启用，继续保留并确保 `decode()` 和 `predict_phenotype()` 中关闭 autocast 以避免 dtype 不匹配。
5. **梯度检查点**：对 HGT/SAGE 卷积层开启 `checkpoint`，以时间换空间。
6. **预训练模型 CPU 推理**：ChemBERTa-2/MolFormer 可在 CPU 上预计算 SMILES 嵌入，训练时仅加载固定向量，避免 GPU 显存占用。
7. **动态 batch size**：HGT 训练 `batch_size` 从 128 降至 64 或 32，SAGE 从 256 降至 128。

### 3.5 8GB 显存下的推荐配置

```yaml
model:
  hidden_dim: 64          # 从 128 降至 64，降低模型容量以换取稳定性
  out_dim: 64
  num_layers: 2
  num_heads: 2            # HGT 头数保持 2
  decoder_type: "residue_bilinear"  # 启用残基解码器，但需配合以下策略

hgt:
  batch_size: 32          # 从 128 降至 32
  num_neighbors: [16, 8]  # 从 [32, 16] 降低

sage:
  batch_size: 128         # 从 256 降低
  num_neighbors: [16, 8]

esm2:
  max_len: 512            # 从 1024 减半
  residue_device: "cpu"   # 强制 CPU 驻留

decoder:
  max_residue_batch: 4    # 从 8 减半
```

**关键结论**：P2 完整 `[L, 640]` 残基特征在当前 8GB 显存和已报告的 CPU RAM 不足场景下风险极高，建议优先采用统计池化特征或大幅降低 `max_len` 与 `max_residue_batch` 后小范围验证。

## 六、综合结论与建议

### 6.1 可行性结论

| 方案                | 可行性     | 优先级 | 关键前提                                                     |
| ----------------- | ------- | --- | -------------------------------------------------------- |
| **P1：HGT 工程修复**   | ✅ 可行    | P0  | 必须首先完成，是所有后续改进的基础                                        |
| **P2：ESM-2 残基特征** | ⚠️ 受限可行 | P1  | 必须解决 8.86 GB 文件加载问题；建议降低 `max_len` 和 `max_residue_batch` |
| **P3：双线性交叉注意力**   | ✅ 可行    | P2  | 在现有 `ResidueAwareBilinearDecoder` 基础上增强                  |
| **P4：预训练分子模型**    | ⚠️ 受限可行 | P2  | 建议 CPU 预计算嵌入，训练时仅使用固定向量                                  |
| **P5：异质图扩展**      | ✅ 可行    | P3  | 在阶段 1-3 验证有效后考虑                                          |
| **课程/DDB/拓扑负采样**  | ✅ 可行    | P1  | 逐步启用，先验证单一策略再组合                                          |
| **完整评估指标体系**      | ✅ 可行    | P0  | 阶段 1 必须完成                                                |

### 6.2 关键决策建议

1. **立即执行阶段 1**：HGT 工程修复和评估完善是当前最高优先级，可在 CPU 环境下完成大部分工作，风险最低。
2. **谨慎推进 P2**：在解决 ESM-2 残基文件加载稳定性问题前，不要强行启用完整 `[L, 640]` 路径。建议先在 `max_len=256` 和 `max_residue_batch=4` 下小批量验证。
3. **推迟 P4 完整集成**：ChemBERTa-2/MolFormer 可在 CPU 预计算后作为固定特征输入，避免 GPU 显存占用过大。
4. **禁止训练时使用** **`add_seed_cpi_edges`**：该参数是信息泄漏高风险点，必须在训练阶段关闭。
5. **早停指标统一为 AUPR**：其他指标仅用于报告，避免早停冲突。

### 6.3 8GB 显存下的推荐落地顺序

1. **第 1 周**：完成 P0 工程修复（维度统一、mini-batch 验证、评估指标、严格拆分）。
2. **第 2 周**：启用拓扑负采样 + 课程负采样，运行完整训练验证指标提升。
3. **第 3-4 周**：小步验证 ESM-2 残基特征（降配参数），若稳定再提升 `max_len`。
4. **第 5-6 周**：尝试预训练分子嵌入（CPU 预计算）。
5. **第 7 周及以后**：考虑 P5 异质图扩展和 P3 高级注意力模块。
